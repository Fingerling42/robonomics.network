## 현대 로봇의 구성 요소

로보틱스에서 작업의 범위를 정의하기 위해, 먼저 로봇이 일반적으로 어떤 부분으로 구성되어 있는지 살펴보겠습니다. 분류 방법은 다양하지만, 현대 로봇의 작업(특히 AI와 밀접하게 연관된 작업)을 고려할 때 다음과 같은 하위 시스템과 그 기능 예시를 들 수 있습니다:

- 구동(예: 모터, 그리핑, 안정화, 균형)
- 센싱(예: 카메라, 라이다, IMU, 센서)
- 정보 처리(예: 인식, 센서 융합, SLAM)
- 제어(예: 관절 제어기, 자세, 동작 최적화)
- 계획(예: 경로, 동작, 작업)
- 추정(예: 오도메트리, 필터, 컨텍스트)
- 통신(예: 네트워크 인터페이스, 인간-기계 상호작용, 음성)
- 추론(예: 기호적 계획, LLM 추론, 논리, 목표 관리, 설명 가능성)
- 학습(예: 딥러닝, 메타러닝, 지속적 학습, 지식 베이스)

## JAM이란 무엇인가?

Polkadot 생태계의 최신 및 향후 혁신을 잘 모르는 분들을 위해 간단히 소개합니다. **JAM**은 네트워크를 분산 슈퍼컴퓨터로 전환하기 위해 설계된 프로토콜입니다. 이더리움과 Polkadot의 강점을 결합하여, 이더리움 스마트 컨트랙트와 유사한 통합된 글로벌 무허가 오브젝트 공간과 Polkadot 고유의 확장성을 갖춘 안전한 병렬 컴퓨팅을 제공합니다.

이는 제안된 것입니다달성 방법은 검증자 컴퓨팅 파워를 가상 **코어**로 분산시키는 것입니다. 무거운 사용자 연산은 특정 코어(*인-코어*)에서만 실행되며, 다른 코어들과 병렬로 처리됩니다. 모든 공유 보안 인프라는 *온체인*에 남아 각 검증자가 실행합니다. 코어에 자신의 코드(**서비스**)를 배포하고자 하는 개발자는 **코어-타임**만 구매하면 되며, 실제로 필요한 자원만큼만 정확히 비용을 지불하면 됩니다.

프로토콜에 대한 자세한 내용은 [JAM Gray Paper](https://graypaper.com/)를 참고하시고, 기계의 예상 특성은 부록에서 확인하실 수 있습니다. 다음으로, JAM이 로보틱스 작업에 어떻게 적용될 수 있는지 살펴보겠습니다.

## JAM의 적용 분야

JAM이 가장 유용할 수 있는 곳은 어디일까요? 가장 명확한 부분부터 제외해봅시다 — 제어 루프 주기가 10ms 미만인 주요 구동 컴포넌트의 연산을 JAM 코어로 옮기는 것은 의미가 없습니다. 이제 개발자들이 그들의 창작물을 어떻게 설명하는지 살펴볼 수 있습니다. Gray Paper에서 Dr. Gavin Wood는 JAM의 개발을 결정짓는 다섯 가지 요소를 제시합니다:

- *탄력성(Resilience)* — 시스템은 참가자의 어떤 개입에도 불구하고 작동을 멈추지 않고 계속 운영되어야 한다
- *범용성(Generality)* — 시스템은 튜링 완전한 연산을 지원해야 한다
- *성능(Performance)* — 시스템은 빠르고 저렴하게 계산을 수행해야 한다
- *일관성(Coherency)* — 시스템은 상태 간의 인과관계를 확립할 수 있어야 한다
- *접근성(Accessibility)* — 누구나 쉽고, 빠르고, 저렴하며, 허가 없이 시스템을 사용할 수 있어야 한다

이 요소들은 이미 JAM 슈퍼컴퓨터가 연산의 검증 가능성과 신뢰 없는 환경에서의 운영이 요구되는 곳에서 특히 유용하다는 힌트를 줍니다. 놀랍지 않게도, 바로 이 지점이 로보노믹스가 창립 이래로 다뤄온 모든 과제 스펙트럼과 완벽하게 맞아떨어집니다:

- 이후 감사 또는 인증을 위한 로봇 행동의 불변 로그
- 장치의 디지털 트윈
- 교환 또는 판매를 위한 텔레메트리의 토큰화
- 로봇 경제: 작업에 대한 결제, “로봇-애즈-어-서비스” 모델

게다가 JAM이 글로벌 연산 환경을 제공하기 때문에, 근본적으로 새로운 솔루션도 생각해볼 수 있습니다. 예를 들어, 코어 내의 디지털 트윈은 단순한 상태 원장(ledger)에서 실제 장치의 스냅샷을 통해 업데이트되는 본격적인 시뮬레이션 모델로 쉽게 변환될 수 있습니다.

JAM은 즉각적인 응답이 필요하지 않은 리소스 집약적인 글로벌 경로 계획을 가능하게 하여, 로봇의 로컬 컴퓨터의 부담을 덜어줄 수 있습니다. 더욱 유망한 것은, 글로벌 수준에서의 작업 최적화입니다., 경로, 그리고 전체 로봇 군집의 목표.

스웜 기술의 경우, 분산 슈퍼컴퓨터는 로봇 간 협업 작업에 완벽하게 적합합니다. 예를 들어, 여러 독립적인 로봇이 환경의 통합 지도를 구축하고 정제하는 협업 신뢰 SLAM, 또는 JAM이 진정한 데이터 분산화와 각 로봇의 학습 기여를 검증하는 연합 학습(신경망 모델의 협업 훈련) 등이 있습니다.

마지막으로, 구동 부품의 저수준에서도 특정 사용 사례가 존재합니다. 예를 들어, 보정 중 PID 컨트롤러 계수를 주기적으로 재계산(이 사실의 기록이 보장됨)하거나, 암호화 서명이 포함된 드라이브용 OTA 펌웨어 업데이트를 제공하는 경우입니다.

물론, 나열된 각 아이디어는 JAM 관점에서 해당 서비스의 논리를 고민해야 합니다. 하지만 개발자들은 이미 **CoreVM**을 준비 중입니다. 이는 운영체제 역할을 하는 특수 서비스로, 개발자에게 지속적인 코드 실행을 위한 컨테이너를 제공하며, 코어의 내부 동작 과정을 감춥니다. 이로써 Polkadot은 Amazon Web Services나 Microsoft Azure와 대체 가능한 분산 클라우드 컴퓨팅 플랫폼으로 변모합니다. 덕분에, 코어 내에서 물리적으로 실행 가능하고 코어-타임이 지불된 경우, 어떤 로보틱스 작업도 실행할 수 있게 됩니다.

특히 주목할 점은 CoreVM에서 음성 합성, 계획 수립, 간단한 대화를 위한 소형 LLM 에이전트를 호스팅할 수 있다는 점입니다. 처음으로, 기업 서비스 대신 분산 클라우드 LLM을 사용할 수 있는 기회가 생겼습니다. 물론, 전용 GPU에서 실행되는 로컬 에이전트가 여전히 더 빠르지만, CoreVM은 전력 소모를 줄여줍니다. 이는 자율성이 몇 시간에 불과한 휴머노이드 로봇에게 매우 중요한 요소입니다.

## 예시: JAM에서 살아가는 디지털 트윈

이제 JAM 코어 내에서 연산이 수행되는 소형 농업용 관수 로봇의 디지털 트윈이 어떻게 살아가는지 시뮬레이션해봅시다. 우리의 목표는 다음 관수 시점을 예측하고, 필요하다면 온도 드리프트로 인한 로봇의 IMU/오도메트리를 재보정하는 것입니다.

1. 매 N분마다, 로봇은 텔레메트리 패키지(토양 수분, 온도, 오도메트리)를 수집하고 **work-package**를 생성합니다. 이때 다음을 명시합니다:
    - 분석 및 보정을 위한 대상 JAM 서비스;
    - 외부 입력으로서의 최신 센서 데이터;
    - 이전 보정 필터 행렬이 저장된 **D3L** 계층(장기 **Distributed, Decentralized, Data Lake**)에서 가져온 세그먼트;
    - 코어-타임 결제 증명 및 기타 파라미터.작업 패키지는 로봇의 키로 서명되어 네트워크로 전송됩니다.

2. 지정된 코어가 패키지를 수신하며, 최소 두 명의 검증자-보증인이 `refine` 함수를 실행합니다:
    - 이전 필터 행렬을 D3L에서 불러옵니다;
    - 새로운 필터를 계산하고 IMU/오도메트리 보정 계수를 얻습니다;
    - 수분이 임계값까지 떨어질 시점을 예측합니다;
    - 예측과 계수를 포함한 **작업-보고서**를 작성합니다;
    - 새로운 필터 행렬을 D3L로 다시 내보냅니다.

    보증인들은 보고서에 서명한 후, 미래 블록의 작성자에게 이를 전송합니다.

3. 작성자가 모든 코어로부터 보고서를 수집한 후, 모든 검증자가 작업-보고서가 올바른지 확인합니다. 그런 다음 `accumulate` 함수가 호출되어:
    - 다음 관개 시간과 보정 계수에 대한 온체인 상태 필드를 업데이트합니다;
    - D3L의 새로운 필터 행렬에 연결합니다.

    이제 남은 것은 블록을 최종화하여 모든 결과가 되돌릴 수 없게 만드는 일뿐입니다.

4. 로봇은 주기적으로 블록체인에서 자신의 상태를 읽고, 새로운 보정 계수를 확인하여 컨트롤러에 적용합니다. 로봇과 그 사용자(예: dapp에서)는 관개 예측을 확인할 수 있습니다.

이제 디지털 트윈의 각 부분이 어디에 존재하는지 설명하면 다음과 같습니다:

- **관개 및 계수의 현재 값** — 온체인 상태
- **필터 행렬** — D3L
- **원시 텔레메트리 패키지** — 작고 오래 필요하지 않은 경우 작업-패키지에 직접 저장; 아카이브되거나 큰 데이터는 감사용으로 D3L에 저장
- **서비스 알고리즘** — 블록체인 상태의 서비스 계정


## 로보틱스의 한계

로보틱스에서 JAM의 사용이 비실용적인 영역이 몇 가지 있습니다. 원칙적으로, 엄격한 실시간 동작, 밀리초 단위의 지연, 새로운 센서 데이터를 즉시 획득해야 하는 로봇 구성요소는 로봇 내에 로컬로 유지하는 것이 더 효율적입니다. 예를 들어, 지속적인 비전 데이터 스트림에 반응하는 모션 플래너나, 중요한 충돌 회피 알고리즘 등이 이에 해당합니다. 센서(3D 라이다, 카메라)에서 나오는 대용량 스트리밍 데이터 처리와 센서 융합은 빠른 처리가 필요하다면 JAM에 너무 무거울 수 있습니다.

요약하자면 — 로보틱스에서 JAM에서의 연산이 정당화되는 경우는 다음과 같습니다:

- 작업의 규모가 수 초 이내에 완료되거나, 완료까지 여러 블록을 기다릴 수 있는 경우
- 블록 최종화를 기다리지 않아도 되는 경우
- 시스템잠재적인 상태 롤백을 허용할 수 있음
- 대용량 데이터의 엄격한 실시간 스트리밍이 필요하지 않거나, 제한된 배치/스냅샷 전송만으로 충분함
- 비교적 안정적인 네트워크 연결이 있음


## 결론

로보틱스 애플리케이션이 없더라도, JAM은 이미 매우 매력적으로 보이며 Web3 기술 스택의 새로운 이정표를 제시합니다. 하지만 지구상의 로보틱스의 미래를 생각해보면, 분산형 슈퍼컴퓨터에서 작동하는 로봇이 다른 어떤 것보다도 더 바람직하고(그리고 덜 두려운) 여러 시나리오를 쉽게 상상할 수 있습니다. 저희 Robonomics 팀의 임무는 이러한 시나리오를 테스트하고 검증된 결과를 커뮤니티에 제시하는 것입니다.

## 부록: 머신의 특성

현재 JAM 그레이 페이퍼(v 0.6.6)에 따르면, 시스템의 특성은 다음과 같습니다:

- 1023개의 검증자 노드
- 단일 검증자 구성: 16코어 CPU, 64GB RAM, 8TB 스토리지, 0.5GbE 네트워킹
- 341코어(1023 / 3 보증인)

프로토콜 지원에 대한 검증자의 부하를 고려할 때, 단일 코어에서 코드에 실제로 사용 가능한 추정 컴퓨팅 파워는 다음과 같습니다:

- 코어 프로세서 파워의 25-50%
- 2GB RAM
- 입출력 대역폭 2MB/s
